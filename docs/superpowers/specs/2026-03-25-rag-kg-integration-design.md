# RAG + 知识图谱集成设计（地震问答项目 v1）

**日期：** 2026-03-25  
**状态：** 已与人脑暴确认范围  
**范围：** 在现有 Neo4j 图谱注入基础上，增加面向 `data/emergency_knowledge.json` 的向量检索（并行混合检索 v1）。

---

## 1. 背景与目标

当前 `app.py` 中 `generate_response` 已通过规则触发 `Neo4jKG` 查询，将结构化地震事件与应急要点拼入提示词，再调用本地微调模型生成回答。该路径对**可枚举实体与关系**有效，但对**用户表述多样、不完全匹配规则**的应急科普问题，缺少基于语义相似度的文档证据。

**v1 目标：**

- 保留现有图谱检索逻辑不变（行为可视为「结构化检索分支」）。
- 新增 **RAG 分支**：仅索引 **`data/emergency_knowledge.json`**，在生成前检索 Top-K 文本块，与图谱上下文**并行**注入同一提示词。
- 论文/答辩可叙述为：**结构化事实（图谱）+ 语义检索（向量）** 的互补，并支持后续消融实验（仅 KG / 仅 RAG / KG+RAG）。

**非目标（YAGNI，v1 不做）：**

- GraphRAG 式社区摘要与全图向量化管线。
- 从用户问题自动抽取实体并做强约束的联合检索（可作为 v2 扩展）。
- 索引 `emergency_knowledge.json` 以外的语料（已确认首轮不做）。

---

## 2. 架构概览

```
用户问题
    ├─► [现有] Neo4jKG：地区 / 震级 / 应急 query_emergency_context 等 → 图谱上下文文本
    └─► [新增] 向量检索：对 emergency_knowledge 分块嵌入 → Top-K → RAG 上下文文本
                        ↓
              提示词拼装（固定顺序：图谱块 → RAG 块 → 问题 → 约束）
                        ↓
              本地微调 LLM 生成
```

两路检索**独立执行**；不在 v1 要求实体对齐，但通过提示词约定冲突时的优先级（见第 5 节）。

---

## 3. 数据与分块策略

**语料文件：** `data/emergency_knowledge.json`

**结构要点（实现依据）：**

- 顶层含 `meta`（说明性文字）、`topics[]`、`topic_relations[]`。**`meta` 不纳入向量索引**，避免与可检索要点混淆。
- `topics[]` 每项含 `id`、`title`、`category`、`source`、`steps[]`（`order`、`text`）。
- `topic_relations[]`：v1 **可不写入向量块正文**，仍由图谱侧表达关系；若实现选择把关系摘要拼入某 topic 块以提升召回，须在实现中注明且控制长度。

**推荐分块（二选一，规划阶段锁定一种即可）：**

- **A（推荐）：** 每个 `topic` 一条检索单元：  
  `title` + `category` + 各 `step.text` 拼接为单一 chunk；元数据：`topic_id`、`source`、`title`。
- **B：** 每个 `step` 一条 chunk，元数据带 `topic_id`、`order`；召回更细但片段更碎，需更高 Top-K 或后处理合并。

**嵌入与向量库（实现选型由计划阶段定，设计层约束）：**

- 中文场景使用轻量嵌入模型（如 bge-small-zh 等），**离线/本地**可运行以匹配当前「本地化部署」定位。
- 向量存储可选 Chroma 本地持久化，或 FAISS + 侧车 JSON 映射（id → 原文与元数据）。索引可在应用启动时构建，或单独脚本构建后加载。

## 4. 与现有代码的边界

| 组件 | 职责 |
|------|------|
| `kg/neo4j_kg.py` | 不变；继续提供结构化查询与应急图谱上下文。 |
| `generate_response`（或等价封装） | 在构建 `prompt` 前：调用 RAG 检索模块，将结果追加为独立段落（如 `【参考资料】`）。 |
| 新增模块（建议） | `rag/` 或 `kg/` 并列的 `rag/emergency_rag.py`：加载/构建索引、`search(query) -> list[{text, topic_id, score}]`，避免 `app.py` 过度膨胀。 |

**配置：** 在 `config/config.py`（或环境变量）中增加：嵌入模型路径、向量库路径、Top-K、每块最大字符数；并设 **独立开关** `KG_CONTEXT_ENABLED`、`RAG_ENABLED`（或等价命名），以支持消融「关 RAG / 关 KG / 全开」而无需改代码路径。

## 5. 提示词与冲突消解

**段落顺序（固定）：**

1. 角色与任务说明（可与现有一致）。
2. `【知识图谱】`（现有 `kg_context`；**无结果时固定写一行** `（无）`；若配置关闭 KG 路径，固定写 `（本路径已关闭）`，与「无结果」区分）。
3. `【参考资料】`（RAG Top-K；**无结果时固定写一行** `（无相关条目）`；RAG 功能关闭时写 `（本路径已关闭）`）。
4. `【问题】` + 用户原文。
5. 回答约束。

**冲突规则（写入系统/用户提示，供模型遵守）：**

- **数值、时间、震级、地点等可验证事实** 以图谱返回为准；若参考资料与图谱矛盾，**以图谱为准**。
- 参考资料仅作**步骤说明、表述补充**；两者皆空时，允许模型基于通用知识作答，但应明确说明「未检索到本地知识库条目」（与现有「无图谱信息」逻辑对齐）。

**长度与截断：**

- 图谱部分维持现有条数/长度策略。
- RAG：Top-K 建议 3～5；单块长度上限与 **`generate_response` 实际使用的 tokenizer `max_length` / 模型上下文** 协调（实现计划阶段对照 `app.py` 或 `llm/serve_earthquake_expert.py` 等调用点核对具体数值，避免文档与代码不一致）。

---

## 6. 错误处理与可观测性

- 向量库未初始化或嵌入加载失败：**降级为仅图谱路径**，日志记录错误；不阻断整个 API（与「本地化可用性」一致）。
- 检索结果为空：不注入虚构内容；提示中说明无参考资料。
- **可选 API debug 字段**（如 `debug: { rag_chunk_ids, kg_query_summary }`）：便于论文消融与答辩演示；默认生产可关。

---

## 7. 测试与验收

- **单元级：** 分块数量、元数据 `topic_id` 正确；给定固定 query，Top-K 包含预期 topic。
- **集成级：** 同一问题在「关 RAG / 关 KG / 全开」三种配置下行为符合预期（消融开关由配置或环境变量控制）。
- **人工抽检：** 若干条「口语化应急问句」在仅规则图谱难触发时，RAG 能召回对应 `topic`。

---

## 8. 通用叙述（论文可用）

知识图谱擅长表达 **实体、关系与可结构化查询的事实**；向量 RAG 擅长覆盖 **表述多样、非结构化的文本证据**。工程上常用 **并行检索 + 提示词分层 + 显式冲突优先级** 组合二者。进阶方向包括 **实体对齐的联合检索** 与 **图衍生摘要索引（GraphRAG）**；本项目 v1 采用轻量并行混合方案以控制复杂度并保留扩展点。

2. **首轮索引语料：** 仅 `data/emergency_knowledge.json`。
